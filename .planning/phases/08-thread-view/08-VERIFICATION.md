## Human Verification Required

The following items cannot be verified programmatically:

### 1. Iframe resize in thread view

**Test:** Open a thread with 3+ messages. Expand 2 messages simultaneously. Verify each email body iframe resizes to its own content height independently.
**Expected:** Each iframe grows to fit its email content; no iframe clips content or overflows into the next message.
**Why human:** `postMessage` height matching by `contentWindow` (e.source) cannot be verified via grep; requires live browser + multiple loaded iframes.

### 2. Collapsible expand/collapse visual behavior

**Test:** On a thread page, click a collapsed message's summary row to expand it. Verify lazy-load fires and email body renders inside the expanded area.
**Expected:** HTMX triggers `hx-get` on the first toggle, fetches `/messages/{id}/body-wrapper`, and replaces the loading placeholder with the email body iframe.
**Why human:** HTMX lazy-load triggered by `<details>` toggle cannot be verified statically; requires browser interaction.

### 3. Thread scroll-to-highlight on load

**Test:** Navigate to `/threads/{id}?highlight={msgId}` where the highlighted message is not the last. Verify the page scrolls to and expands that message with a cyan left border.
**Expected:** The highlighted message is expanded and visible on screen within ~200ms of page load; a cyan left border distinguishes it.
**Why human:** `scrollIntoView` behavior and 150ms setTimeout delay cannot be verified statically.

### 4. `t` key navigation from message detail

**Test:** On any message detail page, press `t`. Verify navigation to the thread view with the correct `?highlight=` parameter in the URL.
**Expected:** Browser navigates (HTMX swap) to `/threads/{conversationId}?highlight={messageId}` and the URL bar updates.
**Why human:** HTMX navigation + `history.pushState` requires a live browser session.
