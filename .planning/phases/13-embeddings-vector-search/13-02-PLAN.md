    - Test 2: RRF with k=60 — message ranked #1 in one list and #5 in other gets score 1/(60+1) + 1/(60+5) = 0.01538 + 0.01538
    - Test 3: RRF deduplicates by message ID — same message in both lists appears once
    - Test 4: RRF returns results sorted by combined score descending
    - Test 5: RRF respects limit parameter
    - Test 6: HybridSearch returns SemanticResult with SimilarityPct from vector component
    - Test 7: When FTS5 returns results but vector returns none, hybrid falls back to FTS5 ranking
    - Test 8: When vector returns results but FTS5 returns none, hybrid uses vector ranking
  </behavior>
  <action>
    1. Create `internal/embedding/hybrid.go`:

    **reciprocalRankFusion(lists [][]rankedItem, k int, limit int) []rankedItem** — Internal helper:
    ```go
    type rankedItem struct {
        MessageID     int64
        SimilarityPct float64 // from vector search, 0 if keyword-only
        RRFScore      float64
    }
    ```
    Algorithm:
    - For each list, assign rank 1..N based on position
    - For each message, accumulate score: `score += 1.0 / (k + rank)`
    - If message appears in vector list, carry its SimilarityPct
    - Sort by RRFScore descending
    - Return top `limit` results

    **HybridSearch(ctx context.Context, client *ai.Client, s *store.Store, engine query.Engine, queryText string, limit int) ([]SemanticResult, error)** — Combines FTS5 + vector:
    a. Run FTS5 search: parse queryText with `search.Parse()`, call `engine.Search(ctx, parsed, limit*2, 0)` (fetch 2x limit for better RRF coverage)
    b. Run semantic search: call `SemanticSearch(ctx, client, s, engine DB, queryText, limit*2)`
    c. Build two rankedItem lists from the results
    d. Call `reciprocalRankFusion(lists, 60, limit)` with k=60 per user decision
    e. Look up full MessageSummary for each result (reuse from either FTS5 or semantic results to avoid extra DB queries)
    f. Return []SemanticResult with SimilarityPct from vector component (0 if message only appeared in FTS5)

    2. Modify `cmd/msgvault/cmd/search.go`:
    - Add `--hybrid` bool flag alongside existing `--semantic` flag
    - When `--hybrid` is set, call `embedding.HybridSearch` instead of pure FTS5 or pure semantic
    - `--hybrid` and `--semantic` are mutually exclusive — error if both set
    - Hybrid output table includes SIMILARITY column (shows "---" for keyword-only matches, percentage for vector matches)
    - JSON output includes `"similarity_pct"` (0 for keyword-only) and `"rrf_score"`

    3. Write tests in `internal/embedding/hybrid_test.go`:
    - Test RRF math with known inputs and expected scores
    - Test deduplication
    - Test fallback when one source returns empty
    - Use table-driven test pattern
  </action>
