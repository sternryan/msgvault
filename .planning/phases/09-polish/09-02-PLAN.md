<objective>
Add loading indicators to all HTMX partial updates across the web UI.

Purpose: POLISH-03 provides visual feedback during every asynchronous operation so users know the app is responding. Without indicators, clicks on pagination, sort headers, tabs, and drill-down rows appear to do nothing until the response arrives.

Output: Every HTMX trigger point in every template has an `hx-indicator` attribute pointing to a nearby indicator span, using the existing `.htmx-indicator` CSS class (yellow text, hidden by default, shown during htmx-request).
</objective>

**Full audit and changes:**

**messages.templ:**
- Message row `<tr>` elements with `hx-get` targeting `#main-content`: add `hx-indicator="#page-indicator"`.

**message.templ:**
- Back to messages link (`hx-get="/messages"`): add `hx-indicator="#page-indicator"`.
- View thread link (`hx-get` to `/threads/{id}`): add `hx-indicator="#page-indicator"`.
- Email toolbar toggle buttons (from Plan 01): these target `#email-body-wrapper` (local swap), so add an inline indicator. Add `<span class="htmx-indicator">Loading...</span>` inside the `.email-toolbar` div (after the toggle buttons/separator area). Point the toggle button `hx-get` elements to `hx-indicator` using the closest approach: give the indicator span an ID like `email-toggle-indicator` and use `hx-indicator="#email-toggle-indicator"` on the toggle buttons. BUT this indicator is inside the swap target (email-body-wrapper), which means it'll be replaced. To avoid this: the body-wrapper handler (from Plan 01) returns the toolbar + body content, and the new response will NOT have the indicator showing (request is done). So the indicator flashing during the request and then being replaced is actually fine — the HTMX request class is added to the trigger element, and `.htmx-request .htmx-indicator` shows the indicator during the request. When the response arrives and replaces the wrapper, the indicator disappears naturally. This works because HTMX shows the indicator on the trigger element (not the target), so as long as the indicator is inside an ancestor that gets the `htmx-request` class, it'll show. Use `hx-indicator="closest .email-toolbar"` pattern — add `class="htmx-indicator"` directly to a loading span inside the toolbar. Actually, simplest: add the `<span class="htmx-indicator">Loading...</span>` inside the toolbar in BOTH the template AND the handler response. That way it exists before AND after the swap.
- "Load images" link: already inside the email-body-wrapper which gets swapped. Same approach — the indicator will flash during request then be replaced by the new content. Add `hx-indicator="closest .email-render-wrapper"` — no, this doesn't work with the standard pattern. Simplest: just let the "Loading..." text in the toolbar serve as the indicator for all body-wrapper operations. Set all body-wrapper trigger elements (toggle buttons, Load images) to `hx-indicator="#page-indicator"` as a fallback. The page indicator is always visible and always persists.

ACTUALLY, let's simplify. For ALL triggers that swap `#main-content` (outerHTML): use `hx-indicator="#page-indicator"`. For ALL triggers that swap LOCAL targets (body-wrapper, stage-result, deletion rows): use `hx-indicator="#page-indicator"` as well. The page indicator is universal and always persists. This is cleaner than per-element indicators and matches the "minimal, data-dense" aesthetic.

Exception: search.templ already has its own `#search-indicator` with "Searching..." text. Keep that — it's better UX than "Loading..." for search.
